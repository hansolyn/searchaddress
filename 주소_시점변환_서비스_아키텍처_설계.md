# 주소 시점 변환 서비스 — DB 스키마 및 아키텍처 설계 문서

버전: v0.1 (초안) · 작성일: 2026-07-07

---

## 1. 문제 정의

**목표**: 사진·영상 기록물의 메타데이터에 기록된 "현재 기준 주소"와 "촬영 시기(연/월)"를 입력받아, 그 시점에 실제로 통용되던 주소를 반환한다.

이 문제는 겉보기와 달리 **단일 API로 해결되지 않는다.** 이유는 한국의 주소·행정구역 체계가 시대별로 완전히 다른 방식으로 관리되어 왔기 때문이다.

| 구간 | 주소 체계 | 데이터 신뢰도 | 자동화 가능성 |
|---|---|---|---|
| 2014 ~ 현재 | 도로명주소 (전면 시행) | 높음 | 대부분 자동화 가능 |
| 1988년경 ~ 2014 | 지번주소 + 법정동코드 체계 | 중간 | 부분 자동화 (코드 체인 추적) |
| ~ 1990년대 이전 | 지번주소, 구조화 데이터 없음 | 낮음 (추정치) | 자동화 불가, 수작업 큐레이션 필요 |

이 설계 문서는 위 세 구간을 **하나의 시계열 데이터 모델**로 통합하되, 구간별로 다른 수집 파이프라인과 신뢰도 라벨을 적용하는 것을 전제로 한다.

## 2. 법적 근거 (최신 기준)

- **도로명주소법**: 시행 2025.2.21, 법률 제20341호
- **도로명주소법 시행령**: 시행 2025.6.21, 대통령령 제35597호 → 2025.8.26 대통령령 제35715호로 재개정 (자전거도로·숲길에도 도로명 부여 가능하도록 도로구간 설정 기준 확대)
- **도로명주소법 시행규칙**: 시행 2022.3.30, 행정안전부령 제324호
- 행정안전부가 **도로명주소법 전부개정법률(안)**을 입법예고한 이력이 확인됨. 전부개정 여부·시행일은 국가법령정보센터(law.go.kr)에서 별도로 최종 확인 필요.
- 참고: https://www.juso.go.kr/CommonPageLink.do?link=/street/lawInfo , https://www.law.go.kr/LSW/lsInfoP.do?lsiSeq=184641

> ⚠️ 법령은 계속 개정되므로, 실제 서비스 오픈 전 `law.go.kr` API 또는 정기 크롤링으로 현행 조문을 재검증하는 배치 작업을 반드시 포함할 것.

## 3. 전체 아키텍처

```
[프론트엔드]
   주소 입력 + 연/월 입력
        │
        ▼
[변환 API 서버] ── (캐시: Redis) ── [행정구역 변천이력 DB (PostgreSQL)]
        │                                     │
        │                          ┌──────────┼──────────┐
        │                          ▼          ▼          ▼
        │                     Tier 1 API   Tier 2 크롤러  Tier 3 큐레이션 테이블
        │                    (juso.go.kr)  (SGIS/법정동)   (수작업 입력 + 출처)
        ▼
[응답: 시점 주소 + 신뢰도(confidence) + 근거 출처 링크]
```

### 3.1 데이터 계층 (Tier)

**Tier 1 — 2014~현재**
- 소스: 행정안전부 실시간 주소정보 조회 API (변동이력여부 파라미터 활용)
- https://www.data.go.kr/data/15057017/openapi.do
- 배치: 월 1회 변경분 동기화

**Tier 2 — 1988년경~2014**
- 소스 ①: 국토교통부 전국 법정동 데이터 (과거법정동코드 필드 포함)
  https://www.data.go.kr/data/15063424/fileData.do
- 소스 ②: 통계청 SGIS 행정구역 경계/코드 API
  https://sgis.kostat.go.kr/developer/html/newOpenApi/api/dataApi/addressBoundary.html
- 소스 ③ (비공식, 보조용): GitHub `vuski/admdongkor` — 연도별 행정동 신설/분할/개칭 변경 로그
  https://github.com/vuski/admdongkor
  - 주의: 공식 정부 데이터가 아니므로 **참고용(confidence=medium)** 으로만 사용하고, 가능하면 공식 소스로 교차 검증

**Tier 3 — 1990년대 이전**
- 공식 API/DB 없음. 공공데이터포털에 유사 요청이 있었으나 행정안전부는 "내부 협의 필요, 별도 신청 절차 이용"으로 답변한 이력 확인됨.
  https://www.data.go.kr/participation/openReqst/OPN_0000000000000344/view.do
- 수작업 소스: 국가기록원 지방행정연혁, 각 시·군 홈페이지 연혁 페이지, 관보(고시) 원문, 지역 향토지
- 운영 방식: 내부 큐레이션 테이블 + 관리자 입력 UI + **출처 URL 필수 첨부** + confidence=low/estimated 라벨 강제

## 4. DB 스키마

### 4.1 핵심 테이블: `region` (행정구역 계통)

같은 지역이 이름/경계가 바뀌어도 하나의 계통으로 추적하기 위한 테이블.

```sql
CREATE TABLE region (
    region_id       BIGSERIAL PRIMARY KEY,
    region_type     VARCHAR(10) NOT NULL,   -- 'sido' | 'sigungu' | 'eupmyeondong' | 'ri'
    parent_id       BIGINT REFERENCES region(region_id),
    legal_code      VARCHAR(10),            -- 법정동코드 (있는 경우)
    created_at      TIMESTAMP DEFAULT now()
);
```

### 4.2 시계열 이력 테이블: `region_history` (SCD Type 2)

```sql
CREATE TABLE region_history (
    history_id      BIGSERIAL PRIMARY KEY,
    region_id       BIGINT REFERENCES region(region_id),
    name            VARCHAR(100) NOT NULL,      -- 예: '조리면' → '조리읍'
    parent_history_id BIGINT REFERENCES region_history(history_id), -- 당시 상위 행정구역
    effective_from  DATE NOT NULL,              -- YYYY-MM-01 로 정규화
    effective_to    DATE,                       -- NULL = 현재까지 유효
    change_type     VARCHAR(20) NOT NULL,       -- '신설'|'개칭'|'통폐합'|'승격'|'편입'|'분할'|'폐지'
    change_note     TEXT,                       -- 통폐합 시 병합 대상 등 자유 서술
    source_url      TEXT NOT NULL,              -- 근거 문서/API 링크 (필수)
    source_tier     SMALLINT NOT NULL,          -- 1 | 2 | 3
    confidence      VARCHAR(20) NOT NULL,       -- 'official' | 'derived' | 'estimated'
    created_at      TIMESTAMP DEFAULT now(),
    verified_by     VARCHAR(50),                -- 수작업 검증자 (Tier 3용)
    verified_at     TIMESTAMP
);

CREATE INDEX idx_region_history_range
    ON region_history (region_id, effective_from, effective_to);
```

### 4.3 도로명주소 이력 테이블 (Tier 1 전용, 2014~)

```sql
CREATE TABLE road_address_history (
    id              BIGSERIAL PRIMARY KEY,
    road_name       VARCHAR(100) NOT NULL,
    building_no     VARCHAR(20),
    region_id       BIGINT REFERENCES region(region_id),
    related_jibun    VARCHAR(50),        -- juso.go.kr 의 관련지번 필드
    effective_from  DATE NOT NULL,
    effective_to    DATE,
    source_api      VARCHAR(50) DEFAULT 'juso.go.kr',
    raw_response    JSONB,               -- 원본 API 응답 보관 (재처리 대비)
    fetched_at      TIMESTAMP DEFAULT now()
);
```

### 4.4 사용자 조회 로그 (품질 개선용)

```sql
CREATE TABLE lookup_log (
    id              BIGSERIAL PRIMARY KEY,
    input_address   TEXT NOT NULL,
    input_year      SMALLINT,
    input_month     SMALLINT,
    matched_confidence VARCHAR(20),
    resolved_tier   SMALLINT,
    no_match        BOOLEAN DEFAULT false,   -- 매칭 실패 시 향후 Tier3 우선순위 산정에 활용
    created_at      TIMESTAMP DEFAULT now()
);
```

## 5. 핵심 조회 로직 (의사코드)

```
function resolveHistoricalAddress(currentAddress, year, month):
    targetDate = normalize(year, month)

    # 1단계: 현재 주소 → 법정동코드 매칭 (juso.go.kr 실시간 API)
    region = matchRegionByAddress(currentAddress)
    if region is None:
        return { status: "주소 매칭 실패", confidence: null }

    # 2단계: region_history 체인을 targetDate 기준으로 역추적
    historyChain = []
    current = region
    while current is not None:
        record = findRegionHistoryAt(current.region_id, targetDate)
        if record is None:
            break
        historyChain.append(record)
        current = record.parent_history_id

    if historyChain is empty:
        return {
            status: "이 시점 데이터 없음 (주로 1990년대 이전)",
            confidence: "unknown",
            suggestion: "수작업 검증 대기열에 등록"
        }

    return {
        address_at_time: buildAddressString(historyChain),
        confidence: min(confidence of all records in chain),  # 체인 중 가장 낮은 신뢰도로 표시
        sources: [r.source_url for r in historyChain]
    }
```

**핵심 원칙**: 체인 중 신뢰도가 가장 낮은 링크가 전체 결과의 신뢰도를 결정한다 (예: Tier1 데이터와 Tier3 추정치가 섞이면 전체 결과는 "estimated"로 표시).

## 7. 신뢰도(Confidence) 표시 정책 — 사용자 대면 원칙

프론트엔드는 반드시 아래 3단계를 색상/배지로 구분해 노출한다.

| confidence | 의미 | UI 표시 |
|---|---|---|
| `official` | 정부 공식 API/DB로 직접 확인됨 | 초록 배지 "공식 확인" |
| `derived` | 공식 코드 체인을 통해 논리적으로 추정 | 노랑 배지 "추정 (근거 있음)" |
| `estimated` | 수작업 리서치, 향토지 등 비공식 소스 기반 | 회색 배지 "참고용 추정치" |

모든 응답에는 **출처 링크가 최소 1개 이상 필수**로 첨부된다 (근거 없는 값은 반환하지 않는다).

## 8. 기술 스택 제안

- **DB**: PostgreSQL (시계열 range 쿼리에 유리, `daterange` 타입 활용 가능)
- **백엔드**: Node.js/Python 무관, 배치 파이프라인은 Python 추천 (공공데이터 CSV/XML 파싱 라이브러리 풍부)
- **캐시**: Redis (juso.go.kr API 호출 결과 캐싱, 승인키 호출 한도 대응)
- **관리자 큐레이션 UI**: Tier 3 수작업 입력을 위한 별도 내부 대시보드 (출처 URL 필수 입력 강제하는 폼 검증 포함)

## 9. 단계별 로드맵 제안

1. **Phase 1**: Tier 1 (2014~현재) 프로토타입 — juso.go.kr API 연동, DB 스키마 중 `region`, `road_address_history` 우선 구현
2. **Phase 2**: Tier 2 (1988~2014) 확장 — 법정동 과거코드 + SGIS 연동, `region_history` 본격 사용
3. **Phase 3**: Tier 3 (1990년대 이전) — 특정 시·군(예: 파주시)부터 수작업 리서치 후 큐레이션 테이블 채움, 이후 지역 확대

## 10. 확인되지 않은 사항 (추가 검증 필요)

- 도로명주소법 전부개정법률(안)의 최종 통과/시행 여부
- Tier 3 구간에 대한 행정안전부 공식 데이터 제공 가능성 (공공데이터 제공신청 절차를 통해 별도 문의 가능)
- SGIS API의 호출 한도 및 상업적 이용 조건 (개발자 등록 후 확인 필요)
